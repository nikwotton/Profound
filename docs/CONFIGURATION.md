# Configuration and secret sources

V0 has one source of truth for each value and deliberately keeps its operator surface small. A value may be delivered to a process as an environment variable, but runtime transport does not make it an operator-configurable deployment knob.

Workload shape and volume, latency and availability SLOs, state-freshness and outage behavior, accounting guarantees, and geography/data-residency or multi-region requirements remain foundational unknowns. Concrete compute, load-balancer, observability, timeout, and retention values are typed v0 implementation defaults used to collect evidence, not stakeholder-approved design commitments.

## Source rules

| Value class               | Source                                                                      | Runtime delivery                                                                                       |
| ------------------------- | --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| Deployed secret           | `sst secret set`; never a deployment environment variable or committed file | SST copies the value into an AWS Secrets Manager secret and injects only the required runtime variable |
| Operator deployment input | Six documented environment-specific network values                          | SST validates and applies the value                                                                    |
| Stage and tuning policy   | Typed code in `infra/stage-config.ts` and `infra/providers/aws.ts`          | SST writes the selected constants into components                                                      |
| CI release wiring         | Release workflow only                                                       | SST receives the immutable image and source revision                                                   |
| Internal component wiring | SST only                                                                    | Generated environment such as service mode, ports, table names, and service URLs                       |
| Test input                | Test command or CI secret/variable                                          | Test-process environment only                                                                          |

## Operator deployment inputs

These are the complete non-secret operator inputs accepted by the v0 SST stack:

| Input                         | Required when                                 | Purpose                                            |
| ----------------------------- | --------------------------------------------- | -------------------------------------------------- |
| `PROXY_DOMAIN`                | Production                                    | Private hostname advertised by proxy access grants |
| `PROXY_CERT_ARN`              | `PROXY_DOMAIN` is set                         | ACM certificate for the TLS proxy listener         |
| `CONTROL_DOMAIN`              | Production                                    | Private HTTPS control-plane hostname               |
| `CONTROL_CERT_ARN`            | Control DNS/certificate is managed externally | ACM certificate for the control plane              |
| `DATA_PLANE_ALLOWED_CIDRS`    | Production; recommended everywhere            | Data-plane network allowlist                       |
| `CONTROL_PLANE_ALLOWED_CIDRS` | Production; recommended everywhere            | Control-plane network allowlist                    |

No token, password, provider credential, timeout, threshold, feature gate, scaling bound, dataset name, or provider capability is accepted through this interface.

## CI release wiring

`RELEASE_IMAGE_URI` and `RELEASE_SHA` are generated by the release workflow. They preserve build-once promotion and deployment traceability; they are not manually tuned stage configuration.

## Typed stage policy

Stage behavior is fixed in `infra/stage-config.ts`:

| Stage                | Provider | Data-plane tasks | Removal          |
| -------------------- | -------- | ---------------- | ---------------- |
| `prod`, `production` | live     | 2–20             | protected/retain |
| `staging`            | live     | 1–4              | removable        |
| `preview`            | mock     | 1–4              | removable        |
| `ci`, `ci-*`         | mock     | 1–4              | removable        |
| personal stage       | mock     | 1–4              | removable        |

The optional multi-identity map, dedicated synthetic route, signed alert destinations, and external usage-accounting source are typed stage features. They are disabled in the initial v0 policy. Enabling one requires a reviewed code change and its corresponding SST secret; there is no environment-variable feature gate.

Only two numeric policy groups are authoritative v0 design decisions: credential lifecycle and the connection-establishment budget. They live together in `src/service-policies.ts` as `V0_POLICY`. Other operational values are current typed implementation defaults in `infra/providers/aws.ts` or component policy modules, including:

- destination ports, connection/operation/stream limits, per-task connection admission, retries, and provider capability policy;
- NLB idle and deregistration timing;
- Axiom endpoint, generated dataset names, and the current telemetry-retention default;
- canary request limits, throttling, and GeoIP accuracy policy;
- health refresh, freshness, cooldown, alert delay, and delivery retries;
- status freshness, notification polling, accounting polling, and reconciliation thresholds.

Changing one of these values changes implementation behavior and therefore goes through code review, tests, and normal deployment rather than an ad hoc environment override. Unless included in `V0_POLICY`, it remains an experimental or unanswered design input rather than a v0 contract.

## SST secret audit

| SST secret                   | Runtime variable                                           | Required when                          | Purpose                                                            |
| ---------------------------- | ---------------------------------------------------------- | -------------------------------------- | ------------------------------------------------------------------ |
| `ControlApiToken`            | `CONTROL_API_TOKEN`                                        | Every non-development deployment       | Authenticates the configured control-plane principal.              |
| `HealthAggregatorToken`      | `HEALTH_AGGREGATOR_TOKEN`; internal control authentication | Every non-development deployment       | Authenticates health evidence and internal health readers.         |
| `CanarySigningSecret`        | `CANARY_SIGNING_SECRET`                                    | Every non-development deployment       | Signs and verifies synthetic canary challenges and rejects replay. |
| `AxiomIngestToken`           | collector-only `AXIOM_TOKEN`                               | Every non-development deployment       | Dataset-scoped telemetry ingestion.                                |
| `BrightDataCustomerId`       | `BRIGHT_DATA_CUSTOMER_ID`                                  | Live-provider stages                   | Confidential provider account identifier.                          |
| `BrightDataZone`             | `BRIGHT_DATA_ZONE`                                         | Live-provider stages                   | Confidential provider routing-zone identifier.                     |
| `BrightDataPassword`         | `BRIGHT_DATA_PASSWORD`                                     | Live-provider stages                   | Bright Data proxy authentication secret.                           |
| `BrightDataApiKey`           | `BRIGHT_DATA_API_KEY`                                      | Live-provider stages                   | Bright Data control/status API credential.                         |
| `ProxidizeApiToken`          | `PROXIDIZE_API_TOKEN`                                      | Live-provider stages                   | Proxidize control API credential.                                  |
| `ControlApiIdentities`       | `CONTROL_API_IDENTITIES_JSON`                              | `stage.features.controlApiIdentities`  | Optional token-to-principal map.                                   |
| `HealthProxyUsername`        | `HEALTH_PROXY_USERNAME`                                    | `stage.features.syntheticHealthRoute`  | Identity half of the dedicated synthetic access grant.             |
| `HealthProxyPassword`        | `HEALTH_PROXY_PASSWORD`                                    | `stage.features.syntheticHealthRoute`  | Secret half of the dedicated synthetic access grant.               |
| `HealthAlertDestinations`    | `HEALTH_ALERT_DESTINATIONS_JSON`                           | `stage.features.healthAlerting`        | Destination URLs plus webhook-signing secrets.                     |
| `UsageAccountingSourceToken` | `USAGE_ACCOUNTING_SOURCE_TOKEN`                            | `stage.features.usageAccountingSource` | Authenticates the optional accounting source.                      |

`sst dev` with a personal stage is forced to mock providers and uses fixed non-sensitive placeholders for its control, health, canary, and disabled Axiom-ingest values. It does not require `sst secret set`. Shared, CI, staging, and production deployments have no placeholders and fail closed when a required secret is missing.

MaxMind account and license credentials are build-time inputs to `pnpm geoip:prepare`, not application runtime secrets. CI supplies them from its secret store; production receives only the prepared database and non-secret metadata. Axiom query tokens and live-provider smoke credentials are test-runner secrets, not application configuration knobs.

## Internal runtime values

SST owns runtime values such as `SERVICE_MODE`, `PROVIDER_MODE`, `ROUTE_TABLE_NAME`, component hosts and ports, advertised proxy addresses, health/service URLs, `CONTROL_API_USER_ID`, `DEPLOYMENT_ID`, OpenTelemetry identity, and every policy value listed above. Operators must not set them independently because doing so can split components from their generated resources or from the reviewed v0 policy. The repository exposes no standalone application environment contract.

The local-only runtime and offline tests inject an ephemeral in-memory `RouteStore`. It intentionally does not claim to verify durability; Dynamo-specific and deployed acceptance tests cover persistence semantics. Deployed services always use DynamoDB.

## Test-only inputs

- Black-box E2E external mode: `E2E_CONTROL_API_URL`, `E2E_CONTROL_API_TOKEN`, `E2E_TARGET_URL`, `E2E_EXPECTED_TARGET_STATUS`. Without `E2E_CONTROL_API_URL`, the runner owns an offline ephemeral local service and controlled recipient.
- Live provider smoke: `RUN_LIVE_PROXY_TESTS` plus provider credentials supplied by CI or a local secret source.
- AWS acceptance: `RUN_AWS_ACCEPTANCE_TESTS`, `DEPLOYED_STAGE`, `DEPLOYED_CONTROL_API_TOKEN`, `DEPLOYED_HEALTH_AGGREGATOR_TOKEN`, `DEPLOYED_CANARY_SIGNING_SECRET`, `DEPLOYED_AXIOM_QUERY_TOKEN`, and documented optional expectations.

Test variables configure the test runner. Application startup and SST infrastructure must not read them.
